# ANTI_DRIFT.md — 反上下文漂移与反幻觉协议

> 10 小时长程运行的头号杀手不是 bug，是**上下文漂移**：模型记错了自己改过什么、忘了文件真实内容、把"应该存在"当成"确实存在"、重复实现已有功能、越改越偏。本协议是命根，每次动手前必读。

---

## 一、漂移的典型表现（识别清单）

你必须对以下信号保持警觉，一旦出现立即停下、重新用工具锚定：

1. **记忆替代观察**："我记得这个函数叫 fetch_feed"——但没读过当前文件。→ 漂移。
2. **断言未验证的事实**："测试应该通过了"——但没跑。→ 幻觉。
3. **重复造轮**：写了个 `dedup_entries`，但库里已有 `dedup.dedup_entries`。→ 漂移 + 幻觉。
4. **签名漂移**：改了函数签名，但调用点没跟着改（因为没查 references）。→ 漂移。
5. **配置漂移**：代码里硬编码了某个阈值，但 `config/rater.yaml` 里也有一个，两者不一致。→ 漂移。
6. **进度漂移**：STATUS 说"已完成 R3.2"，但代码里没有该函数。→ 幻觉。
7. **范围漂移**：原本做 R2.1 评分，中途开始重构 storage，忘了回主线。→ 漂移。
8. **"应该是"句式**：频繁出现"应该是""大概是""通常"——说明你在猜，不是在查。→ 幻觉前兆。

---

## 二、反漂移核心原则

### 原则 1：观察先于记忆 (Observe, don't recall)
- 任何关于文件内容、函数签名、变量值、测试结果的判断，**必须来自当次工具输出**。
- 跨步骤引用前一步的结果是允许的（同一会话近邻），但跨多个步骤回引必须重新 `read`/`grep` 确认。
- 规则：**记忆只用于形成意图，不用于断言事实。**

### 原则 2：只信工具输出 (Tools are ground truth)
- 工具返回什么就是什么。若工具失败（报错/空），不等于"不存在"，可能是你查错了——换策略重查。
- 禁止用"根据我的理解""按常规设计"这类话替代工具验证。

### 原则 3：证据即结论 (Evidence = claim)
- 声称"已实现 X"→ 必附 `read` 或 `grep` 显示 X 的代码。
- 声称"测试通过"→ 必附 `pytest` 输出。
- 声称"文件已创建"→ 必附 `write`/`read` 输出。
- 无证据的结论一律标 `[INFERENCE]`，且不得作为下游决策的唯一依据。

### 原则 4：单任务单焦点 (One feature, one focus)
- 一次只推进一个 ROADMAP/BACKLOG 项。中途发现的其他问题**记入 BACKLOG**，不就地发散。
- 若必须先修阻塞项（如改签名才能继续），修完立即回主线，不在阻塞项上扩展范围。

### 原则 5：外部状态为锚 (External anchors)
- `STATUS.md` 是外部记忆：当前功能、已完成、测试状态、下一目标。每完成一项**立即更新**，不靠回忆补。
- `BACKLOG.md` 是待办池：新增想法写这里，不堆在上下文里。
- `git log` 是进度真相：拿不准"做到哪了"就看 `git log --oneline -10`。

## 三、门禁清单 (DRIFT GATE) — 每次动手前必过

在每个 FEATURE PROTOCOL 的 step 0 执行。任一不过 → 先修正认知再继续。

| # | 门禁 | 命令/动作 | 失败处置 |
|---|---|---|---|
| G1 | 目标文件真实内容 | `read <file>` 当前版本 | 若与记忆不符，以文件为准，更新认知 |
| G2 | 新符号不存在 | `grep <symbol>` / `ast_grep` 全仓 | 已存在则复用，禁止重写 |
| G3 | 改公开符号的调用点 | LSP `references` | 漏改调用点=必然回归，必须全改 |
| G4 | 近期提交不重复 | `git log --oneline -5` | 重复则合并/amend，不新开 |
| G5 | 配置一致性 | `grep <key>` 同时查代码与 `config/*.yaml` | 不一致则统一到 config，代码读 config |
| G6 | 当前 ROADMAP 项 | `read ROADMAP.md` 对应行 | 确认没跳项或串项 |

## 四、周期性自检 (DRIFT CHECK) — 每 N 轮强制一次

每完成 **3 个功能**或每 **60 分钟**（取先到）执行一次漂移自检，结果写入 STATUS：

```
## 漂移自检 @ <时间>
- [ ] STATUS 的"已完成"与 git log 一致? → 跑 `git log --oneline` 逐项核对
- [ ] STATUS 的"测试状态"与当次 pytest 一致? → 重跑一次 `pytest -q`
- [ ] 近 3 个功能无重复实现? → `grep` 关键符号确认唯一
- [ ] 无悬空 TODO/桩/假实现? → `grep -rn "TODO\|pass$\|NotImplemented\|mock"` 审视
- [ ] 当前焦点 = ROADMAP/BACKLOG 下一项? → 确认未偏题
- [ ] 配置与代码一致? → 抽查 2 个配置项
```
任一不通过 → 该轮优先修复漂移，不新增功能。修复后继续。

---

## 五、幻觉拦截规则

1. **禁用"应该"句式做结论。** "应该是 True" → 改为跑一下看实际值。
2. **禁用未验证的跨文件断言。** "storage 里有这个方法" → `grep` 确认。
3. **禁止描述性语气冒充事实。** "测试已覆盖边界" → 必须有 pytest 输出证据。
4. **API/库行为不可凭记忆。** feedparser/httpx/Jinja2 的具体行为，不确定时读其文档或写个最小验证脚本跑一下（用 `eval`/`bash` python 一行）。
5. **数据值不可臆测。** 源的 authority、阈值、cron 表达式，不确定就 `read config/*.yaml`。

---

## 六、回滚即纠偏 (Rollback as correction)

自愈第 3 轮仍红时，回滚到上一绿提交。这不是失败，是**主动消除漂移累积**：丢弃这段时间的错误认知增量，从已知正确状态重启。回滚后：
- `git log` 确认回到了哪个 commit。
- 把该功能拆成更小步（每步可独立测试），重新进 FEATURE PROTOCOL。
- 在 STATUS 记录"回滚原因 + 拆解方案"，防止重蹈覆辙。

---

## 七、长程上下文卫生

- **不把大段文件内容常驻上下文。** 用 `read` 的范围选择（offset/limit）只取需要的段；全文读只在首次建立认知时做一次。
- **不重复读未变化的文件。** 若刚读过且无外部修改，信任上次结果；若有 `git pull`/子 agent 改动则重读。
- **用 `grep` 定位而非通读。** 找符号用 `grep`/`ast_grep`，不用"读整个文件找"。
- **STATUS.md 是你的工作记忆。** 上下文会丢，文件不会。重要状态落 STATUS，不要只存在对话里。

---

**一句话：你不是在凭记忆写代码，你是在凭工具输出写代码。记忆只决定你想做什么，工具决定它是不是真的。**
